# Threading the Gate Through a Mastra Pipeline

Exact source patches for wiring the QA step into a shilpiworks-style codebase using the `createAgentRunner` factory pattern. If your codebase has different conventions, adapt the principles — the key invariants are at the bottom of this file.

## Prerequisite

You have:
- A Mastra workflow file (`<agent>-workflow.js`) chaining `<quote-step> → build-prompt → generate-image-gemini`
- The `createAgentRunner` factory pattern in `agent-runner.js`
- A `publishProduct` function in `publish.js` that writes a Prisma row + uploads to a blob store
- A new file `<agent>-steps.js` that exports `<agent>VisualQAStep` (generated by `scripts/scaffold-qa-gate.mjs`)

## Patch 1: workflow file

Splice the QA step into the chain. The `.map` between `generate-image-gemini` and the QA step is what injects `expectedQuote` from the upstream quote step (the image-gen step doesn't carry it forward).

```diff
 import {
   <agent>QuoteStep,
   build<Agent>ImagePromptStep,
+  <agent>VisualQAStep,
 } from './<agent>-steps.js';

 export const <agent>Workflow = createWorkflow({...})
   .then(<agent>QuoteStep)
   .then(build<Agent>ImagePromptStep)
   .map(...)
   .then(generateImageStepGemini)
+  // Splice in the QA gate. The .map below pulls expectedQuote from the
+  // upstream quote step (generate-image-gemini doesn't carry it forward)
+  // and passes through every field the publish step expects.
+  .map(async ({ inputData, getStepResult }) => {
+    const quoteOut = getStepResult('<quote-step-id>');
+    return {
+      base64: inputData.base64,
+      responseId: inputData.responseId,
+      analysis: inputData.analysis,
+      textError: inputData.textError,
+      attempt: inputData.attempt,
+      retryFeedbacks: inputData.retryFeedbacks,
+      openaiImageRequests: inputData.openaiImageRequests,
+      openaiCorrectionRequests: inputData.openaiCorrectionRequests,
+      expectedQuote: quoteOut?.quote || '',
+    };
+  })
+  .then(<agent>VisualQAStep)
   .commit();
```

Replace `<agent>` with the agent name (e.g. `lorignite`) and `<quote-step-id>` with the upstream step's id (e.g. `lorignite-quote`).

## Patch 2: agent-runner.js (factory)

The factory invokes `publishProduct` after the workflow completes. After the QA step is in the workflow, `imageResult.qaResult` is populated. Thread it through.

```diff
       const published = await publishProduct({
         base64: imageResult.base64,
         analysis,
         type,
         theme: resolvedTheme,
         style,
         designPrompt: promptResult?.designPrompt || '',
         author: quoteResult?.author || null,
         contextHint: promptResult?.context || '',
         brand,
         watermarkText,
+        qaResult: imageResult.qaResult,
       });
```

Single line. The factory itself doesn't need to know what `qaResult` is — it just passes it through.

## Patch 3: publish.js

`publishProduct` accepts the new option, sets `isActive: false` on kill, and persists the verdict.

```diff
-export async function publishProduct({ base64, analysis, type, theme, style, designPrompt, author, contextHint, brand, watermarkText }) {
+export async function publishProduct({ base64, analysis, type, theme, style, designPrompt, author, contextHint, brand, watermarkText, qaResult }) {
   const db = getPrisma();
   // ...

   // Build features JSON
   const features = {
     ai_content: { ... },
     agent_generated: true,
     agent_type: type,
     agent_theme: theme,
     agent_style: style,
   };
   if (author) features.quote_author = author;
   if (brand) features.brand = brand;
+  if (qaResult) features.qa_verdict = qaResult;
+
+  // QA gate: kill-class verdicts land the product as inactive (soft-kill).
+  // The Product still gets created — preserves the image + verdict for
+  // admin review, instead of throwing and losing the evidence.
+  const initialIsActive = qaResult?.severity === 'kill' ? false : true;

   // ...

   product = await db.product.create({
     data: {
       // ...
-      isActive: true,
+      isActive: initialIsActive,
     },
   });
```

If your `Product.features` Prisma column is `String` (JSON-stringified, not native `Json`), the existing code likely has `features: JSON.stringify(features)` later — that handles `qa_verdict` automatically without further changes.

## Patch 4: storefront filter (already there in most cases)

Verify the brand's collection page filters by `isActive: true`:

```js
const products = await prisma.product.findMany({
  where: { isActive: true, ... },
  // ...
});
```

If it doesn't, add it. Otherwise QA-killed products will leak onto the storefront.

## Patch 5: admin review surface (optional but recommended)

To review QA-killed products and selectively flip them back to active, add an admin route or a small script. Minimum viable:

```js
// scripts/list-qa-killed.js
import { getPrismaClient, disconnect } from './lib/db.js';
const p = getPrismaClient();
const rows = await p.product.findMany({
  where: { isActive: false, features: { contains: '"qa_verdict"' } },
  select: { id: true, name: true, imageUrl: true, features: true },
  orderBy: { createdAt: 'desc' },
  take: 50,
});
for (const r of rows) {
  const f = JSON.parse(r.features);
  console.log(`[${r.id}] ${r.name}`);
  console.log(`  img: ${r.imageUrl}`);
  console.log(`  verdict: ${f.qa_verdict.severity} — ${f.qa_verdict.failures.join('; ')}`);
}
await disconnect();
```

To override a false positive:
```js
await prisma.product.update({ where: { id: <productId> }, data: { isActive: true } });
```

## Key invariants (for non-shilpiworks codebases)

If your stack is different, port the principles, not the patches:

1. **The QA step must NEVER throw on KILL.** If your workflow framework treats step errors as workflow errors, you'll lose the image. Always resolve successfully with the verdict attached.

2. **The verdict must reach the persistence layer.** Whoever writes the row/file/record needs to know the severity. Thread `qaResult` through every layer between the QA step and the writer.

3. **KILL = inactive flag, not deletion.** The image must persist for admin review and false-positive recovery. Don't `rm` the file or skip the DB write.

4. **The verdict must be readable later.** Persist enough detail (which items failed, what the model OCR'd, what was expected) that you can audit false-positive rate after a few weeks of data.

5. **The storefront/consumer filter must respect inactive.** If the surface that shows products to users doesn't filter by active, the gate's protection is invisible.
